# JS / Node Internals

## What is Node.js?

- JavaScript is a language spec; to execute it, you need an engine — V8 is Chrome's engine
- Node.js = V8 engine + system APIs (files, network, processes) + event loop
- Node lets JavaScript run outside the browser
- npm handles dependency management, versioning, and script running

## Threading Model

- JS runs on a single thread: 1 call stack, 1 event loop, task queues
- Event-driven, non-blocking I/O model
- Perfect for I/O-heavy workloads (DB calls, API calls, network waits) — handles thousands of connections cheaply
- Not suited for CPU-heavy work — a long computation blocks the single thread and starves other requests

## Event Loop & Task Queues

- Microtasks run **before** the next macrotask — this is a common interview question
- **Microtasks:** `Promise.then`, `queueMicrotask`
- **Macrotasks:** `setTimeout`, `setInterval`

## libuv

- Node uses libuv under the hood for cross-platform async I/O
- Provides: event loop, thread pool (default 4 threads), OS async APIs
- Thread pool handles file system, DNS, some crypto — not JS execution
- Flow: JS thread submits work → thread pool executes → callback queued back to JS thread

## V8 Engine

- Parse JS → generate bytecode → JIT-compile hot paths into machine code
- Frequently called functions get optimized ("hot path")
- JS is **not** purely interpreted — it JIT-compiles at runtime
- Dynamic types can cause **de-optimization**: stable types stay fast, changing types force V8 to bail out of optimized code

## Types & Quirks

- `typeof null === "object"` — a historical bug in JS, kept for backwards compatibility
- JS has automatic garbage collection via mark-and-sweep

## Prototypes & Classes

- JS was designed as **prototype-based**, not class-based — `class` syntax is syntactic sugar over prototypes
- No classical inheritance; objects delegate to prototype chains
- `this` depends on **how** a function is called, not where it's defined — a frequent source of bugs and interview questions
- JS is multi-paradigm: OOP via prototypes, functional, imperative

## Closures

- A closure is a function that retains access to its **lexical environment** (variables in scope where it was written)
- Closures are like "functions with memory"
- Enable: private/encapsulated state, callbacks with context, async logic

```js
function outer() {
    let x = 10;
    return function inner() {
        return x; // inner closes over x
    };
}
```

## Language Spec vs Implementations

- ECMAScript is the spec; V8, SpiderMonkey, JavaScriptCore are implementations (like C++ spec → gcc/clang)
- Different browsers have different engine internals, but all conform to the spec

## Architecture Note (AlgoLens)

- BM25/TF-IDF ranking is compute-heavy — keep it in a C++/Go service
- Node handles API + orchestration; CPU work goes to the ranking service
- Node event loop must stay responsive — CPU-heavy ranking would block it
- Can use gRPC to bridge Node ↔ C++ service
