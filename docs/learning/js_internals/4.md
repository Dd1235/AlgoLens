# JS Interview Q&A

## Explain the JS Event Loop

- JS is single-threaded but handles async tasks via the event loop
- Blocking ops (timers, network) are offloaded to Web APIs (browser) or libuv (Node)
- When done, their callbacks are placed in the **task queue** (macrotasks) or **microtask queue**
- Event loop rule: call stack must be empty → drain all microtasks → pick one macrotask → repeat
- This is why `setTimeout(fn, 0)` still runs after synchronous code and after any pending `.then` callbacks

## Explain Closures

- A closure is when a function retains access to variables from its outer scope, even after that outer function has finished executing
- The inner function holds a reference to the lexical environment, not a copy of values

```js
const createCounter = () => {
    let count = 0;
    return () => {
        count += 1;
        return count;
    };
};

const counter = createCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(count);     // ReferenceError — count is private to createCounter's scope
```

- Use cases: private state, factory functions, memoization, callbacks with context

## What is Hoisting?

- JS engine allocates memory for declarations before executing code — this is hoisting
- `var` declarations are hoisted and initialized to `undefined` — accessible before the line, but value is undefined
- `let` / `const` are hoisted but sit in a **temporal dead zone** — accessing before declaration throws `ReferenceError`
- Function **declarations** are fully hoisted — can be called before they appear in the file
- Function **expressions** (`const fn = () => {}`) are NOT hoisted — the variable is hoisted but the function isn't assigned yet

```js
console.log(a);   // undefined (var hoisted)
console.log(b);   // ReferenceError (let in TDZ)
greet();          // works — function declaration hoisted
hello();          // TypeError — hello is undefined at this point

var a = 1;
let b = 2;
function greet() { return "hi"; }
const hello = () => "hello";
```

## What is the Purpose of Anonymous Functions?

- Functions without a name, used inline where a named function isn't needed
- Keep code concise; make the callback's intent visible at the call site
- Common in higher-order functions, event handlers, and one-off callbacks

```js
arr.filter(x => x >= 0);
btn.addEventListener("click", () => { /* ... */ });
```

- Downside: anonymous functions show as `(anonymous)` in stack traces, harder to debug

## What is Debounce?

- Debounce delays a function call until after a specified pause in invocations — only fires once the user "stops"
- Example: search input — don't fire an API request on every keystroke, wait until the user pauses typing

```js
function debounce(fn, delay) {
    let timer;
    return (...args) => {
        clearTimeout(timer);
        timer = setTimeout(() => fn(...args), delay);
    };
}
```

- Use cases: search/autocomplete, form validation, window resize handlers, preventing double-clicks

## How Do You Handle Errors in async/await?

- Wrap `await` calls in a `try/catch` block — rejected promises throw like synchronous errors

```js
async function fetchData() {
    try {
        const res = await fetch(url);
        const data = await res.json();
        return data;
    } catch (err) {
        console.error("Failed:", err);
    }
}
```

- Alternatively, chain `.catch()` on the returned promise from the async function
- Always handle rejections — unhandled promise rejections crash Node processes in newer versions
