# Node.js Single-Threaded Model

## The Problem with Multi-Threading

- Traditional servers (e.g. Apache HTTP) spawn one thread per request — doesn't scale under high concurrency
- Excessive context switching between threads wastes CPU
- Each thread consumes memory even when idle

## Single-Threaded Execution

- JS uses a **call stack**: functions are pushed on call, popped on return
- Only one thread executes JS at a time — no parallelism for JS code itself
- A synchronous blocking operation (e.g. a sync network request) freezes the entire thread

## Event-Driven Non-Blocking I/O

- Node uses an **event loop** to delegate I/O tasks instead of blocking
- Flow: request (e.g. read a file) → event loop → **event demultiplexer** monitors the I/O op → notifies event loop when done → callback is queued and processed
- The event demultiplexer is the OS-level mechanism (epoll on Linux, kqueue on macOS) that watches multiple I/O sources efficiently

## libuv

- libuv (C++ library) powers Node's async I/O — it decides how each task should be handled
- Truly async ops (network sockets) → handed to OS async APIs, no thread needed
- CPU-bound or blocking ops (file system, DNS, some crypto) → offloaded to **thread pool** (default 4 threads)
- Thread pool result → callback queued back onto the event loop → JS picks it up

## When Single-Threading Breaks Down

- Great for I/O-heavy workloads: DB calls, API calls, file reads — the thread stays free while waiting
- Bad for CPU-intensive work (image processing, heavy computation) — blocks the event loop and starves all other requests
- Solution: offload CPU work to worker threads (`worker_threads` module) or a separate service (C++/Go)
