# How JavaScript Code Runs

## Execution Pipeline

- Source code → **tokenization** (keywords, operators, literals, identifiers) → **AST** (Abstract Syntax Tree) → **JIT compilation** → execution
- Parsing produces an AST, not direct machine code — the AST is what the compiler works from
- JIT (Just-In-Time) means compilation happens at runtime, not ahead of time
- After execution: garbage collection (mark-and-sweep, runs periodically) and the event loop (continuously checks call stack + message queue)

## JS Engine vs Runtime

- A JS **engine** (V8, SpiderMonkey, JavaScriptCore) handles parsing, compiling, and executing JS
- A JS **runtime** wraps the engine and adds APIs — browsers add DOM/fetch/setTimeout, Node adds fs/http/process
- Node.js embeds V8 (a C++ program) and exposes server-side APIs — same engine, different runtime

## Dev vs Production

**Dev:**
- Local machine, live server, browser hits files directly

**Production:**
- Files are bundled and minified (optimize payload size)
- Deployed to a remote server
- Browser requests page → receives HTML → fetches CSS + JS → JS engine begins execution

## Other Runtimes

- Phaser: browser-based game framework, runs in the browser runtime
- Deno, Bun: alternative server-side JS runtimes (also built on V8 or JavaScriptCore)
