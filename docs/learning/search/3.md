# Search Index

## Why Not a Regular Database?

- Standard DB indexes (B-tree) work on exact values or prefixes — not full-text search
- Text can match at any position in a string, or across multiple fields
- A `LIKE '%keyword%'` query does a full table scan — doesn't scale
- Search needs a fundamentally different data structure: the **inverted index**

## Inverted Index

- Tokenize all documents → for each token, store a list of document IDs (and optionally positions)
- Structure: `token → [doc_id, doc_id, ...]`
- Lookup is O(1) for exact token match — just hash or tree lookup on the token

```
"graph"   → [doc3, doc7, doc12]
"search"  → [doc1, doc3, doc9]
```

## Prefix Search

- Keep tokens sorted (e.g. in a trie or sorted index)
- Prefix queries ("algo\*") become a binary search or trie traversal — fast

## Suffix Search

- Reverse each token at index time: "apple" → "elppa"
- Suffix queries ("_berry") become prefix queries on the reversed index: search "yrreb_"
- Allows "ends with" queries without full scans

## Apache Lucene

- The foundational library behind Elasticsearch and Solr
- Built on an **LSM tree** (Log-Structured Merge-tree) — optimized for high write throughput
- Writes go to an in-memory buffer (memtable), flushed to immutable **segments** on disk
- Segments are periodically merged — merging re-sorts and compacts the inverted index
- This is why Elasticsearch is eventually consistent on search: newly indexed docs aren't visible until the segment is flushed and refreshed

## Index vs Database — When to Use Which

- Use a search index (Elasticsearch, Typesense, Meilisearch) when you need full-text search, fuzzy matching, or relevance ranking
- Use a database when you need transactions, joins, or exact lookups
- Common pattern: store source of truth in a DB, sync to a search index for query serving

## Inverted Index

- Break text into words/tokens
- Lowercasing, removing punctuation, stemming/lemmatization
- Remove common words (stopwords)
